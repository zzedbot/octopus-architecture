# ⚠️ 失效处理机制 (Failure Handling)

**版本**: v1.0  
**生效日期**: 2026-03-10

---

## 一、概述

失效处理机制确保八爪鱼架构在异常情况（主脑离线、触手故障、网络中断等）下能够安全暂停、恢复和继续执行。

---

## 二、失效场景分类

| 场景 | 严重性 | 影响范围 | 处理方式 |
|------|--------|----------|----------|
| **主脑离线** | 🔴 高 | 全部触手 | 暂停待命，状态持久化 |
| **单触手故障** | 🟡 中 | 单个触手 | 任务转移或重新分配 |
| **吸盘失败** | 🟢 低 | 单个子任务 | 重试或跳过 |
| **网络中断** | 🟡 中 | 通讯受阻 | 本地缓存，恢复后同步 |
| **Channel 故障** | 🟡 中 | 通讯不透明 | 降级执行，记录日志 |

---

## 三、主脑离线处理

### 3.1 检测机制

```
主脑离线检测流程：

1. 触手监测心跳
   ↓
2. 90 秒未收到心跳
   ↓
3. 确认主脑离线
   ↓
4. 触发失效处理
```

### 3.2 处理流程

```
┌─────────────────────────────────────────────────────────┐
│  触手检测到主脑离线                                      │
└────────────────────┬────────────────────────────────────┘
                     │
                     ↓
┌─────────────────────────────────────────────────────────┐
│  步骤 1: 保存当前状态                                     │
│  - 保存到 tentacle-state/current-task.json              │
│  - 创建历史快照                                          │
└────────────────────┬────────────────────────────────────┘
                     │
                     ↓
┌─────────────────────────────────────────────────────────┐
│  步骤 2: 暂停当前任务                                     │
│  - 停止吸盘生成                                          │
│  - 等待进行中的吸盘完成                                  │
│  - 标记任务为 paused 状态                                │
└────────────────────┬────────────────────────────────────┘
                     │
                     ↓
┌─────────────────────────────────────────────────────────┐
│  步骤 3: 发送告警                                        │
│  - 通过外部 Channel 发送告警消息                       │
│  - 消息："[⚠️] 主脑离线，已暂停待命"                       │
└────────────────────┬────────────────────────────────────┘
                     │
                     ↓
┌─────────────────────────────────────────────────────────┐
│  步骤 4: 进入等待状态                                     │
│  - 定期监测心跳恢复                                      │
│  - 不执行新任务                                          │
│  - 保持低功耗运行                                        │
└─────────────────────────────────────────────────────────┘
```

### 3.3 告警消息格式

```json
{
  "type": "alert",
  "level": "critical",
  "tentacle": "Eva",
  "timestamp": "2026-03-10T15:00:00+08:00",
  "event": "brain_offline",
  "message": "[⚠️] 主脑离线，已暂停待命",
  "details": {
    "lastHeartbeat": "2026-03-10T14:58:30+08:00",
    "offlineDuration": 90,
    "currentTask": "爬取文档 V3 章节",
    "taskProgress": 60
  }
}
```

## 四、主脑恢复流程

### 4.1 恢复步骤

```
1. 主脑重新上线
   ↓
2. 立即发送心跳
   ↓
3. 触手收到心跳，退出暂停状态
   ↓
4. 主脑请求状态汇报
   ↓
5. 触手从 tentacle-state/ 恢复状态
   ↓
6. 触手汇报当前状态
   ↓
7. 主脑确认，任务继续
```

### 4.2 状态汇报格式

```json
{
  "type": "status_report",
  "tentacle": "Eva",
  "timestamp": "2026-03-10T16:00:00+08:00",
  "status": "paused",
  "currentTask": {
    "taskId": "task-20260310-001",
    "taskName": "爬取文档 V3 章节",
    "progress": 60,
    "pausedAt": "2026-03-10T15:00:00+08:00",
    "pausedReason": "brain_offline"
  },
  "readyToResume": true
}
```

### 4.3 恢复确认格式

```json
{
  "type": "resume_command",
  "from": "Zed",
  "to": "Eva",
  "timestamp": "2026-03-10T16:01:00+08:00",
  "command": "resume",
  "taskId": "task-20260310-001",
  "message": "系统恢复，任务继续"
}
```

## 五、单触手故障处理

### 5.1 故障检测

| 检测方式 | 说明 |
|----------|------|
| **心跳无响应** | 触手多次未响应主脑查询 |
| **任务超时** | 任务超过预计时间未完成 |
| **状态异常** | 触手状态文件损坏或无法读取 |

### 5.2 处理流程

```
1. 主脑检测到触手故障
   ↓
2. 标记触手为 offline 状态
   ↓
3. 检查故障触手的任务
   ↓
4. 决策：
   ├─ 任务可转移 → 重新分配给其他触手
   ├─ 任务不可转移 → 等待触手恢复
   └─ 任务可取消 → 取消任务并通知主人
   ↓
5. 发送告警到外部 Channel
```

### 5.3 任务转移策略

| 任务类型 | 转移策略 |
|----------|----------|
| **文档处理** | 可转移给 Eva 或其他有空闲触手 |
| **代码开发** | 可转移给 Leo |
| **数据分析** | 可转移给 Anna |
| **设计任务** | 可转移给 Mia |
| **协调管理** | 由 Geoff 接管或主脑直接处理 |

## 六、吸盘失败处理

### 6.1 失败分类

| 失败类型 | 处理方式 |
|----------|----------|
| **临时错误**（网络超时） | 自动重试（最多 3 次） |
| **永久错误**（资源不存在） | 标记失败，跳过或人工干预 |
| **部分失败**（部分数据获取失败） | 记录已获取数据，标记失败部分 |

### 6.2 重试策略

```javascript
async function executeWithRetry(task, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await task.execute();
    } catch (error) {
      if (i === maxRetries - 1) {
        // 最后一次重试失败
        throw error;
      }
      
      // 等待后重试（指数退避）
      const delay = Math.pow(2, i) * 1000; // 1s, 2s, 4s
      await sleep(delay);
    }
  }
}
```

## 七、网络中断处理

### 7.1 检测机制

| 症状 | 判定 |
|------|------|
| Channel 发送失败 | 网络可能中断 |
| 心跳延迟过高 | 网络质量差 |
| 部分触手失联 | 局部网络问题 |

### 7.2 降级策略

```
网络中断时的降级策略：

1. 本地缓存消息
   - 缓存到 /{tentacle-id}/workspace/message-queue/
   
2. 继续执行任务（如可能）
   - 不依赖外部资源的任务继续执行
   - 依赖外部资源的任务暂停
   
3. 定期重试连接
   - 每 30 秒重试一次
   
4. 恢复后同步
   - 网络恢复后，发送缓存消息
   - 同步任务状态
```

## 八、Channel 故障处理

### 8.1 故障影响

| 影响 | 说明 |
|------|------|
| **通讯不透明** | 主人无法看到主脑↔触手通讯 |
| **命令接收延迟** | 主人命令可能无法及时到达 |
| **状态汇报丢失** | 任务完成可能无法及时通知 |

### 8.2 降级策略

```
Channel 故障时的处理：

1. 记录日志
   - 所有通讯记录到本地日志文件
   
2. 继续执行
   - 不阻塞任务执行
   
3. 尝试备用 Channel
   - 如配置了备用 Channel，切换使用
   
4. 恢复后补发
   - Channel 恢复后，补发重要消息
```

## 九、监控与告警

### 9.1 监控指标

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| `brain.offline.count` | 主脑离线次数 | > 3 次/小时 |
| `tentacle.failure.count` | 触手故障次数 | > 5 次/小时 |
| `sucker.failure.rate` | 吸盘失败率 | > 10% |
| `heartbeat.latency` | 心跳延迟 | > 5 秒 |
| `state.save.errors` | 状态保存错误 | > 3 次/小时 |

### 9.2 告警级别

| 级别 | 颜色 | 说明 | 通知方式 |
|------|------|------|----------|
| **Critical** | 🔴 | 主脑离线、系统不可用 | 立即通知主人 |
| **Warning** | 🟡 | 单触手故障、任务失败 | 外部 Channel 消息 |
| **Info** | 🔵 | 吸盘失败、重试 | 日志记录 |

## 十、恢复检查清单

### 10.1 主脑恢复检查清单

```
主脑重新上线后的检查清单：

□ 1. 发送心跳
□ 2. 请求所有触手状态汇报
□ 3. 检查触手响应情况
□ 4. 确认未响应的触手（如有）
□ 5. 决策：等待/转移任务/取消任务
□ 6. 发送恢复确认给触手
□ 7. 通知主人系统恢复
□ 8. 记录失效事件到日志
```

### 10.2 触手恢复检查清单

```
触手恢复执行前的检查清单：

□ 1. 读取 tentacle-state/ 状态文件
□ 2. 验证状态有效性
□ 3. 检查任务是否过期
□ 4. 汇报状态给主脑
□ 5. 等待主脑确认
□ 6. 恢复任务执行
□ 7. 记录恢复事件到日志
```

## 十一、测试用例

### 11.1 主脑离线测试

```
给定：系统正常运行
当：主脑突然离线
则：
  1. 所有触手在 90 秒内检测到离线
  2. 所有触手保存状态到 tentacle-state/
  3. 所有触手进入 paused 状态
  4. 发送告警到外部 Channel
```

### 11.2 主脑恢复测试

```
给定：主脑离线，触手暂停
当：主脑重新上线
则：
  1. 主脑发送心跳
  2. 触手收到心跳，退出暂停
  3. 触手汇报状态
  4. 主脑确认，任务继续
```

### 11.3 触手故障测试

```
给定：Eva 正在执行任务
当：Eva 突然故障（无响应）
则：
  1. 主脑标记 Eva 为 offline
  2. 主脑检查 Eva 的任务
  3. 任务重新分配给其他触手（如可能）
  4. 发送告警到外部 Channel
```
