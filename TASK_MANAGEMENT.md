## 一、任务生命周期

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   pending   │ ──→ │ in_progress │ ──→ │ completed   │
│   (待处理)   │     │   (执行中)   │     │   (已完成)   │
└─────────────┘     └──────┬──────┘     └─────────────┘
                           │
                           ↓
                     ┌─────────────┐
                     │   paused    │
                     │   (暂停)     │
                     └──────┬──────┘
                            │
                            ↓
                     ┌─────────────┐
                     │   failed    │
                     │   (失败)     │
                     └─────────────┘
```

## 二、任务结构

### 2.1 任务定义

```json
{
  "taskId": "task-20260310-001",
  "taskName": "爬取文档 V3 章节",
  "description": "从外部文档平台爬取 V3 章节的所有文档",
  "assignedBy": "Zed",
  "assignedTo": "Eva",
  "assignedAt": "2026-03-10T14:30:00+08:00",
  "deadline": "2026-03-11T18:00:00+08:00",
  "priority": "normal",
  "status": "in_progress",
  "progress": 60,
  "subtasks": [
    {
      "id": "sucker-1",
      "name": "爬取第 1-5 章",
      "status": "completed",
      "result": "5 篇文档"
    },
    {
      "id": "sucker-2",
      "name": "爬取第 6-10 章",
      "status": "in_progress",
      "result": null
    },
    {
      "id": "sucker-3",
      "name": "爬取第 11-15 章",
      "status": "pending",
      "result": null
    }
  ],
  "metadata": {
    "source": "外部文档平台",
    "url": "https://vip.kingdee.com/knowledge/specialDetail/218022218066869248"
  }
}
```

### 2.2 任务字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `taskId` | string | ✅ | 任务唯一标识 |
| `taskName` | string | ✅ | 任务名称 |
| `description` | string | ❌ | 任务详细描述 |
| `assignedBy` | string | ✅ | 分配者（主脑） |
| `assignedTo` | string | ✅ | 执行者（触手 ID） |
| `assignedAt` | datetime | ✅ | 分配时间 |
| `deadline` | datetime | ❌ | 截止时间 |
| `priority` | string | ❌ | 优先级 (high/normal/low) |
| `status` | string | ✅ | 任务状态 |
| `progress` | number | ❌ | 进度百分比 (0-100) |
| `subtasks` | array | ❌ | 子任务（吸盘）列表 |
| `metadata` | object | ❌ | 元数据 |

## 三、任务分配

### 3.1 分配流程

```
1. 主人发送命令
   ↓
2. 主脑解析任务
   ↓
3. 主脑选择合适的触手
   │  - 根据任务类型
   │  - 根据触手负载
   │  - 根据触手专业性
   ↓
4. 主脑通过 sessions_send 分配任务
   ↓
5. 触手确认接收
   ↓
6. 触手开始执行
```

### 3.2 触手选择策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **专业匹配** | 选择最匹配任务类型的触手 | 专业任务（文档、代码、设计等） |
| **负载均衡** | 选择当前空闲的触手 | 通用任务、杂项任务 |
| **轮询分配** | 轮流分配给各触手 | 均衡负载，避免单点过载 |

### 3.3 优先级处理

| 优先级 | 处理策略 |
|--------|----------|
| **high** | 立即分配，可打断低优先级任务（需主脑决策） |
| **normal** | 按正常流程分配 |
| **low** | 等待触手空闲时分配 |

## 四、任务执行

### 4.1 单触手执行

```
触手接收任务
   ↓
评估任务复杂度
   ↓
┌──────────────────────┐
│  任务是否简单？       │
└──────────┬───────────┘
           │
    ┌──────┴──────┐
    │             │
   是            否
    │             │
    ↓             ↓
直接执行    拆分为吸盘
    │             │
    │        ┌────┴────┐
    │        │吸盘 1    │
    │        │吸盘 2    │
    │        │吸盘 3    │
    │        └────┬────┘
    │             │
    └──────┬──────┘
           │
           ↓
    汇总结果
           │
           ↓
    汇报给主脑
```

### 4.2 多触手协作

```
主脑接收复杂任务
   ↓
分解为多个子任务
   ↓
┌───────────────────────────────┐
│ 分配给多个触手                 │
│ ┌───────┐  ┌───────┐  ┌──────┐│
│ │ Eva   │  │ Leo   │  │ Mia  ││
│ │文档   │  │代码   │  │设计  ││
│ └───┬───┘  └───┬───┘  └──┬───┘│
└─────┼──────────┼──────────┼────┘
      │          │          │
      └──────────┼──────────┘
                 │
                 ↓
          主脑汇总结果
                 │
                 ↓
          汇报给主人
```

## 五、任务监控

### 5.1 进度汇报

触手应定期向主脑汇报任务进度：

| 场景 | 汇报频率 |
|------|----------|
| **长任务** (>1 小时) | 每 30 分钟 |
| **中任务** (15-60 分钟) | 每 15 分钟 |
| **短任务** (<15 分钟) | 完成时汇报 |
| **关键节点** | 立即汇报 |

### 5.2 进度消息格式

```
[{触手} → Zed] 进度：{任务描述} ({百分比}%)
- 已完成：{完成内容}
- 进行中：{当前工作}
- 预计完成：{预计时间}
```

**示例**:
```
[Eva → Zed] 进度：爬取文档 V3 章节 (60%)
- 已完成：第 1-10 章，共 10 篇文档
- 进行中：第 11-15 章爬取中
- 预计完成：2026-03-10T17:00:00+08:00
```

## 六、任务完成

### 6.1 完成汇报

```
[{触手} → Zed] ✅ 任务完成：{任务名称}
- 结果：{结果描述}
- 输出：{输出文件/位置}
- 耗时：{执行时长}
```

**示例**:
```
[Eva → Zed] ✅ 任务完成：爬取文档 V3 章节
- 结果：成功获取 45 篇文档
- 输出：/eva/workspace/output/v3-chapters/
- 耗时：2 小时 30 分钟
```

### 6.2 结果归档

任务完成后，触手应：

1. 保存最终状态到 `tentacle-state/`
2. 归档任务历史
3. 清理临时文件
4. 更新为 idle 状态

## 七、任务异常处理

### 7.1 异常分类

| 异常类型 | 处理方式 |
|----------|----------|
| **任务失败** | 记录原因，汇报主脑，等待指示 |
| **任务超时** | 汇报主脑，申请延期或取消 |
| **资源不足** | 汇报主脑，申请资源或转移任务 |
| **主脑离线** | 保存状态，暂停待命 |

### 7.2 失败汇报格式

```
[{触手} → Zed] ❌ 任务失败：{任务名称}
- 原因：{失败原因}
- 已尝试：{已尝试的解决方法}
- 建议：{建议的后续处理}
```

**示例**:
```
[Leo → Zed] ❌ 任务失败：构建部署包
- 原因：依赖冲突，npm install 失败
- 已尝试：清理 node_modules，更新 npm
- 建议：需要升级 Node.js 版本或调整依赖
```

## 八、任务状态持久化

### 8.1 持久化时机

| 时机 | 说明 |
|------|------|
| **任务接收** | 保存任务基本信息 |
| **状态变更** | 任务状态变化时 |
| **定期保存** | 每 5 分钟自动保存 |
| **主脑离线** | 紧急保存 |

### 8.2 持久化位置

```
/{tentacle-id}/workspace/tentacle-state/
├── current-task.json     # 当前任务状态
├── progress.json         # 进度信息
└── context.json          # 上下文信息
```

详细规范见：[STATE_PERSISTENCE.md](./STATE_PERSISTENCE.md)
